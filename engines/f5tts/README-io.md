# F5-TTS engine — synthesis plan and I/O contract

This module is a **scaffold, not a verified synthesizer**. This document is the single place
that records what we know about the real F5-TTS ONNX pipeline (with sources), what this engine
implements against that knowledge, and exactly what remains unverified. Every fact below that
isn't directly quoted/cited from an upstream source is marked `ASSUMPTION`.

F5-TTS (`SWivid/F5-TTS`, CFM/flow-matching DiT, Apache-2.0) is a **voice-cloning** model: it needs
a short **reference audio clip + the exact transcript of that clip**, and generates new speech in
that voice by iteratively denoising noise conditioned on (reference audio, reference text, text to
speak). It has no "speaker embedding table" the way Piper/Kokoro do — cloning IS the mechanism.

## Sources consulted

- **`DakeQQ/F5-TTS-ONNX`** (GitHub, community ONNX export/inference port of `SWivid/F5-TTS`):
  - `Export_ONNX/F5_TTS/Export_F5.py` — the `torch.onnx.export(...)` calls that fix the three
    graphs' literal input/output tensor names and `dynamic_axes`.
  - `Export_ONNX/F5_TTS/F5-TTS-ONNX-Inference.py` — the reference Python inference driver: session
    creation, the NFE denoising loop, text preprocessing, WAV writing.
- **`SWivid/F5-TTS`** (HuggingFace `SWivid/F5-TTS`, upstream weights/vocab) — `F5TTS_Base/vocab.txt`
  and `F5TTS_v1_Base/vocab.txt` (both 13,800 bytes — same vocabulary), and
  `src/f5_tts/model/utils.py` (`list_str_to_idx`, `get_tokenizer`, `convert_char_to_pinyin`) for the
  real text→id contract.
- Two independent HF re-uploads of the ONNX export (`huggingfacess/F5-TTS-ONNX`,
  `CreativeHub008/F5TTS-onnx`) confirm the exported file set is always exactly
  `F5_Preprocess.onnx` + `F5_Transformer.onnx` + `F5_Decode.onnx` (+ `vocab.txt` in at least one of
  the two) — this is the bundle fingerprint [inspect()][F5TtsEngine.inspect] relies on.

No ONNX graph was actually downloaded/run in this environment (multi-hundred-MB weights, no ORT
native runtime here) — everything below is from reading the exporter/inference *source code*, not
from executing it.

## Pipeline: three ONNX graphs, one Python (here: Kotlin) orchestration loop

```
reference audio (wav) ─┐
reference text  ───────┼─► F5_Preprocess.onnx ─► noise, rope_cos_q/sin_q/cos_k/sin_k,
text to speak  ────────┘                          cat_mel_text, cat_mel_text_drop, ref_signal_len
                                                        │
                                                        ▼
                                    ┌─── F5_Transformer.onnx (called NFE-1 times) ───┐
                                    │   in:  noise, rope_*, cat_mel_text(_drop),      │
                                    │        time_step                               │
                                    │   out: denoised, time_step                     │
                                    │   denoised(t) becomes noise(t+1); loop         │
                                    └──────────────────────────────────────────────────┘
                                                        │  (after the last step)
                                                        ▼
                                          F5_Decode.onnx (Vocos vocoder)
                                          in:  denoised, ref_signal_len
                                          out: output_audio
```

### Graph 1 — `F5_Preprocess.onnx`

Source: `Export_F5.py`, the first `torch.onnx.export(f5_preprocess, (refaudio, text_ids,
max_duration), ..., input_names=['audio', 'text_ids', 'max_duration'], output_names=['noise',
'rope_cos_q', 'rope_sin_q', 'rope_cos_k', 'rope_sin_k', 'cat_mel_text', 'cat_mel_text_drop',
'ref_signal_len'], dynamic_axes={...})`. These are the literal ONNX tensor names (quoted, not
inferred) — same situation as `PiperEngine`'s `input`/`input_lengths`/`scales`, i.e. a **stable,
citable export contract**, not exporter-autogenerated names.

| tensor | role | `ASSUMPTION` dtype/shape (ORT default for this shape of op; not graph-verified) |
|---|---|---|
| `audio` (in) | reference-clip waveform | float32 `[1, 1, ref_samples]`, normalized `[-1, 1]` (matches this app's `FloatArray` convention throughout, see `core/audio/Pcm16.kt`) |
| `text_ids` (in) | vocab ids of `ref_text + gen_text` concatenated | int64 `[1, T]` — the port's own script reads/writes this as int32 in numpy before feeding ORT; this engine's `Tensor` seam only has float/long, so this is fed as `Tensor.longs` — `ASSUMPTION`, needs on-device confirmation that the graph accepts int64 |
| `max_duration` (in) | target mel-frame count for the whole utterance — this **is** F5's native duration/speed knob (see below) | int64 scalar `[1]` |
| `noise` (out) | initial ODE state | float32 `[1, max_duration, mel_dim]` |
| `rope_cos_q`/`rope_sin_q`/`rope_cos_k`/`rope_sin_k` (out) | rotary position embeddings for the transformer's attention | float32, shapes carrying `max_duration` per `dynamic_axes` |
| `cat_mel_text` (out) | conditioned (text-attached) hidden state | float32 `[1, max_duration, ...]` |
| `cat_mel_text_drop` (out) | **unconditioned** hidden state (CFG's "text dropped" branch — confirmed by `Export_F5.py`'s `text_drop = self.f5_text_embed(...)`, concatenated with zeros) | float32, same shape as `cat_mel_text` |
| `ref_signal_len` (out) | reference-audio length in mel frames, needed by the decoder to trim the reference portion back out of the generated mel | int64 scalar |

### Graph 2 — `F5_Transformer.onnx` (the flow-matching DiT step, called in a loop)

Source: `Export_F5.py`'s second export, `input_names=['noise', 'rope_cos_q', 'rope_sin_q',
'rope_cos_k', 'rope_sin_k', 'cat_mel_text', 'cat_mel_text_drop', 'time_step']`,
`output_names=['denoised', 'time_step']`.

**CFG is baked into this graph, not a runtime input.** `Export_F5.py` instantiates
`F5Transformer(f5_model, cfg=CFG_STRENGTH, steps=NFE_STEP, ...)` and the forward pass computes
`noise += (pred + (pred - pred1) * self.cfg_strength) * self.delta_t[time_step]` — both the
conditioned (`pred`, from `cat_mel_text`) and unconditioned (`pred1`, from `cat_mel_text_drop`)
passes happen inside ONE `session.run()`, combined with a `cfg_strength` that was frozen into the
graph at export time (`CFG_STRENGTH = 2.0` in the exporter). There is **no runtime CFG-scale
input** to route a UI knob to.

**NFE step count is also baked in, not runtime-discoverable.** The exporter precomputes
`t = torch.linspace(0, 1, steps)` and a `time_expand` tensor sized to `steps` (`NFE_STEP`,
default 32) *inside `F5Transformer.__init__`*, and that becomes an ONNX graph constant. The
Python inference driver's own loop —
`for i in range(0, NFE_STEP - 1, FUSE_NFE): noise, time_step = ort_session_B.run(...)` — runs
exactly `NFE_STEP - 1` times to match whatever step count that specific exported `.onnx` file was
built with; changing "quality vs. speed" upstream means **exporting a different
`F5_Transformer.onnx`**, not flipping a runtime parameter. Our `InferenceSession`/`Runtime` seam
(`core/runtime/*`) has no graph-metadata introspection (no way to ask an opened session "how many
steps do you expect"), so this engine cannot *discover* the right loop count from the loaded
graph — it uses `NFE_STEP - 1` = 31 (the DakeQQ default) as an internal, documented constant
(`TRANSFORMER_STEPS` in `F5TtsEngine`), **not exposed as a `ModelParameter`**.

This is why this engine declares **no NFE/quality `ModelParameter`**: CLAUDE.md rule 1 requires a
tunable to be *discovered*, not assumed, and neither NFE nor CFG is a value the loaded graph
exposes for us to read back — both are frozen into whichever `F5_Transformer.onnx` file the user
downloaded. A future improvement could read ONNX `metadata_props`/`doc_string` if the exporter
ever records the baked step count there, or extend `Runtime` with a metadata-query method; neither
exists today. (Mirrors the honesty of `CosyVoice2Engine`, which also declares zero parameters
because its native ABI exposes no knob — see its KDoc.)

### Graph 3 — `F5_Decode.onnx` (Vocos vocoder)

Source: `Export_F5.py`'s third export, `input_names=['denoised', 'ref_signal_len']`,
`output_names=['output_audio']`. Takes the last transformer iteration's `denoised` mel plus
`ref_signal_len` (to slice the reference-audio portion of the mel back off) and produces the
final waveform. `ASSUMPTION`: `output_audio` is treated as float32 PCM in `[-1, 1]` (the app-wide
convention, `core/audio/Pcm16.kt`) — the reference Python script writes it via
`sf.write(..., format='WAVEX')`, which is compatible with either float or int16 source data, so
the exact on-graph dtype is not nailed down here.

## Speed: routes to `max_duration`, not resampling (CLAUDE.md rule 2)

The reference driver computes:

```
max_duration = ref_audio_len + int(ref_audio_len / ref_text_len * gen_text_len / SPEED)
```

(`ref_audio_len`/`gen_text_len` etc. quoted from `F5-TTS-ONNX-Inference.py`; `ref_audio_len` is in
mel frames, i.e. `ref_samples / HOP_LENGTH` with `HOP_LENGTH = 256`). `max_duration` is a **real,
declared input** of `F5_Preprocess.onnx` (`input_names=[..., 'max_duration']`) that fixes how many
mel frames the whole pipeline generates — a smaller `max_duration` at the same text length means
the model must speak faster to fit, and vice versa. That is a genuine model-native duration
control (like Piper's `length_scale`), so this engine **does** declare a `speed` `ModelParameter`
and implements this exact formula in `F5DurationPlanner`, never resampling output audio.

## Voice = a bundled reference clip (not a baked speaker table)

`VoiceEngine.synthesize(text, voiceId, params)` has no channel for the caller to hand over
arbitrary reference-audio bytes at call time, and `Voice` (`core/engine/Voice.kt`) is just
`{id, name, language}`. So, consistent with how every other engine's "voices" are things the
*bundle* ships (Piper's per-speaker `sid`s, CosyVoice3's baked voice bank), a "voice" here is a
**reference clip bundled alongside the three graphs**, discovered at `inspect()` time by this
engine's own naming convention:

- `<name>.reference.wav` + `<name>.reference.txt` (exact transcript of the clip) → one `Voice`
  with id `<name>`.

**This convention is invented by this engine, not an upstream standard** — DakeQQ's port takes
`--reference_audio`/`--reference_text` as ad-hoc CLI flags, there is no packaging convention for
"ship N reference clips with the model". If a bundle has zero such pairs, `inspect()` still claims
it (the three graphs + `vocab.txt` are the real fingerprint) and the descriptor falls back to a
single voice id `"reference"` with no backing files — `doLoad()` succeeds (the ONNX sessions open
fine), but `synthesizeSentence()` fails loudly (fail-closed, rule 4) because there is nothing to
clone. **TODO**: a real reference-audio-driven voice picker (record/import a clip + type its
transcript in the UI, not just bundle-shipped pairs) is out of scope here and left for a follow-up
issue.

## Tokenizer / `vocab.txt`

`SWivid/F5-TTS`'s `list_str_to_idx` (`src/f5_tts/model/utils.py`, quoted): `torch.tensor([
vocab_char_map.get(c, 0) for c in t])` — **one entry per Unicode character**, index = the
character's 0-indexed line number in `vocab.txt`, and any character absent from the vocab maps to
index `0` (confirmed to be intentional: `get_tokenizer` "validates that space maps to index 0",
and `vocab.txt`'s first line is a literal space). `F5Vocab` in this module reproduces exactly that:
parse `vocab.txt` into a `Map<String, Int>` keyed by line content (line index = id, SSOT — nothing
hardcoded), then map each input character through it with the same `getOrDefault(c, 0)` fallback.

**Not implemented**: `convert_char_to_pinyin` — real F5-TTS Mandarin input goes through jieba
segmentation + `pypinyin` (TONE3-style) before hitting `list_str_to_idx`; `vocab.txt` is full of
pinyin-syllable rows (`ai1`, `ai2`, ...) for exactly this reason. This module has no Kotlin
jieba/pypinyin equivalent, so Chinese-script input is passed through character-by-character (each
hanzi will almost always miss the vocab and fall back to id `0`, i.e. silently degrade rather than
error — matches rule 4's fail-closed spirit at the character level, but the **output will be
wrong** for Mandarin text). **TODO**, documented, not fabricated.

## What this engine implements vs. leaves TODO

| Piece | Status |
|---|---|
| `inspect()`/`forcedMatch()` fail-closed fingerprint (3 onnx files + `vocab.txt`) | Implemented + tested |
| Descriptor (sample rate 24000, `speed` parameter, voices, asset paths, resource cost) | Implemented + tested |
| `F5Vocab` (vocab.txt parsing, char→id, OOV→0) | Implemented + tested (pure, deterministic) |
| `F5DurationPlanner` (`max_duration` formula) | Implemented + tested (pure, deterministic) |
| `F5WavDecoder` (mono 16-bit PCM WAV → normalized `FloatArray`, for bundled reference clips) | Implemented + tested (pure, deterministic) |
| Reference-clip voice discovery from bundle | Implemented + tested |
| Three-session orchestration (`doLoad`, NFE loop shape, tensor key wiring) | Implemented per the cited export contract, `ASSUMPTION`-marked at every dtype/shape not nailed down by source, **not run against a real graph** |
| Mandarin `convert_char_to_pinyin` | **TODO** — not implemented |
| NFE step count as a runtime knob | **Deliberately not exposed** — baked into the graph, undiscoverable via the current `Runtime` seam (see above) |
| Live (non-bundled) reference-audio recording UI | **TODO** — out of scope |

Bottom line: everything deterministic (parsing, formulas, fingerprinting, descriptor building) is
implemented and unit-tested on the JVM with no ONNX runtime. The three-graph orchestration is
wired end-to-end and structurally sound against the cited export contract, but — like the rest of
this codebase's `ASSUMPTION` markers — it has not been run against real F5-TTS ONNX weights and
needs on-device verification before anyone trusts its output.
